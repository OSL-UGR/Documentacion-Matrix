
El archivo de configuración de Nginx se encuentra dentro del siguiente path:

- `/etc/nginx/conf.d/matrix.ugr.es-ok_20260416.conf`

Su función es recibir todo el tráfico web externo desde Internet, aplica seguridad a través de los certificados SSL y decidir que contenedor de docker interno debe enviar cada petición.

Este archivo tiene la siguiente forma:

``` Nginx
# /etc/nginx/conf.d/matrix.conf

# -------------------------------------------------------------------
# BLOQUE 1: Redirección HTTP a HTTPS
# -------------------------------------------------------------------
server {
    listen 80;
    listen [::]:80;
    server_name matrix.ugr.es;
    
    # Redirige todo el tráfico HTTP no seguro a HTTPS
    return 301 https://$host$request_uri;
}

# -------------------------------------------------------------------
# BLOQUE 2: Servidor principal HTTPS
# -------------------------------------------------------------------
server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name matrix.ugr.es;

    # Certificados SSL Institucionales (CSIRC UGR)
    # Requisito: Los archivos deben existir y tener permisos de lectura para el usuario nginx
    ssl_certificate /etc/pki/tls/certs/fullchain.pem;
    ssl_certificate_key /etc/pki/tls/private/privkey.pem;

    # Logs específicos del servicio
    access_log /var/log/nginx/matrix.access.log; 
    error_log /var/log/nginx/matrix.error.log;

    # Tamaño máximo para subida de adjuntos (Imágenes, documentos)
    client_max_body_size 50M;

    # Cabeceras de seguridad HTTP
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header X-Content-Type-Options nosniff;
    add_header X-Frame-Options SAMEORIGIN;
    add_header X-XSS-Protection "1; mode=block";

    # -------------------------------------------------------------------
    # RUTAS DE PROXY INVERSO
    # -------------------------------------------------------------------

    # 1. Backend (Synapse API)
    location ~* ^(\/_matrix|\/_synapse\/client) {
        proxy_pass http://127.0.0.1:8008;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        
        # Extensión del timeout: Crítico para el long-polling de /sync en Matrix
        proxy_read_timeout 600; 
    }

    # 2. Frontend (Element Web)
    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # Soporte para WebSockets
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }

    # -------------------------------------------------------------------
    # DESCUBRIMIENTO (Well-known)
    # -------------------------------------------------------------------
    
    # Autoconfiguración para clientes móviles y web
    location /.well-known/matrix/client {
        default_type application/json;
        return 200 '{"m.homeserver": {"base_url": "https://matrix.ugr.es"}}';
        add_header Access-Control-Allow-Origin *;
    }

    # Descubrimiento para Federación de servidores
    location /.well-known/matrix/server {
        default_type application/json;
        return 200 '{"m.server": "matrix.ugr.es:443"}';
        add_header Access-Control-Allow-Origin *;
    }
}
```

## Explicación de la Configuración

La estructura del archivo está dividida por bloques dependiendo de la funcionalidad:

### Bloque 1: Cifrado Obligatorio (HTTP a HTTPS)

Este bloque de servidor escucha en el puerto 80, por lo que interfecpta tráfico no cifrado. Si un usuario intentase entrar al matrix a través de `http://matrix.ugr.es`, lo expulsamos y lo redirigimos automáticamente a la versión segura de `https://`.

### Bloque 2: Servidor Principal y Seguridad (HTTPS)

Este es el bloque que gestiona el tráfico seguro con el puerto 443 y el protocolo HTTP/2. Dentro de este configuramos:

- **Certificados institucionales:** Para ello utilizamos las rutas oficiales con los certificados  `fullchain.pem` y `privkey.pem` oficiales de la Universidad de Granada, asegurando que los navegadores reconozcan el servidor como una entidad legítima sin alertas de seguridad.
	
- **Límite de subida:** `client_max_body_size 50M` limita a los usuarios enviar archivos mayores a 50 Megabytes. Esto lo hacemos para que los usuarios puedan adjuntar estos, o enviar fotos y vídeos libremente ya que la limitación por defecto si no ponemos nada de Ngynx bloquearía las subidas mayores a 1MB.
    
- **Headers de Seguridad:** Esto es un poco por seguridad y convención, inyectamos reglas estrictas para prevenir vulnerabilidades web comunes.

### Rutas de Proxy Inverso (Enrutamiento Interno)

Aquí es donde Nginx actúa como puerta hacia los contenedores de Docker. Es por eso que tendremos que configurar tanto el Backend como el Frontend

- **Backend (Synapse)**: Se encarga de detectar cualquier solicitud que empiece por `/_matrix` o `/_synapse/client`, las encargadas de Synapse, por eso las redirigimos al puerto interno 8008. Además de esto es muy importante poner el timeout:
	
	Hemos añadido la directiva `proxy_read_timeout 600.` A diferencia de la navegación web normal, el protocolo Matrix utiliza "long-polling" que es una sincronización para los mensajes en tiempo real. Esto implica que Element abre una conexión y Synapse la mantiene abierta y en espera todo el rato hasta que alguien envíe un mensaje. Por defecto Nginx asume que cualquier conexión que se encuentre inactiva durante más de 60 segundo es un error. Es por eso que tenemos que subir ese tiempo a 600 (unos 10 minutos) ya que si no,  los usuarios sufrirían cada minuto de micro-desconexiones constantes en la interfaz, además de que podemos saturar el servidor a peticiones.
	
- **Fronted (Element):** todo el tráfico general, por eso ponemos `/` se envía al cliente Element en el puerto 8080. Además de esto es muy importante activar el tráfico para los WebSockets:
	
	 Tenemos que añadir las directivas `Upgrade` y `Connection "upgrade"`. Por defecto, la web funciona sobre el protocolo HTTP estándar, donde el cliente simplemente pide datos y el servidor responde cerrando la conexión. Sin embargo para funciones propias del chat que nos piden que haya una inmediatez bidireccional absoluta, necesitamos abrir un túnel permanente  llamado WebSocket. Si omitiéramos esas líneas, Ngynx bloquearía por defecto la creación del túnel, rompiendo por completo la sincronización en tiempo real de mensajería y demás otras funciones.

### Bloque de Descubrimiento (Well-Known)

Este bloque es vital para el funcionamiento en red del protocolo Matrix y la comodidad de los usuarios. Responde con pequeños archivos JSON estructurados:

- **Federación (`/server`):** Permite que otros servidores del mundo (ej. `matrix.org`) pregunten: _"¿A qué puerto envío los mensajes para los usuarios de la UGR?"_, y Nginx responde delegando al puerto seguro `443`.
    
- **Clientes (`/client`):** Cuando un usuario descarga la App móvil de Element y escribe `@usuario:matrix.ugr.es`, la aplicación consulta esta ruta para descubrir automáticamente que la base de la API está en `https://matrix.ugr.es`, sin que el usuario deba configurarlo manualmente.
