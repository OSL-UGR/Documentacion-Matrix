Este script de Bash ubicado en `/home/svc/backup_matrix.sh` se encarga de realizar copias de seguridad de los datos críticos del servidor. Si en algún momento la base de datos se corrompiese o se perdiesen los datos, siempre se podrán recuperar gracias a este script.

Este script está configurado para ejecutarse todos los días a las 3AM de forma desatendida mediante una tarea  cron. El código de script tiene la siguiente forma:


```bash
#!/bin/bash

DIRECTORIO_MATRIX="/home/svc"
DIRECTORIO_BACKUPS="$DIRECTORIO_MATRIX/backups"
FECHA=$(date +"%Y-%m-%d_%H-%M")

# Nos situamos en el directorio
cd $DIRECTORIO_MATRIX

# Tenemos que cerrar el servicio, ya que si la base de datos se actualiza al mismo tiempo que realizamos el backup, podríamos corrromperla
sudo docker compose stop synapse

# Comprimimos la base de datos, los ficheros de configuración y las claves de cifrado
sudo tar -czvf $DIRECTORIO_BACKUPS/backup_synapse_$FECHA.tar.gz ./data/homeserver.db ./data/homeserver.yaml ./data/*.key ./data/*.crt ./element-config.json

# Reabrimos el servicio de synapse
sudo docker compose start synapse

# Borramos los backups que lleven más de una semana 
find $DIRECTORIO_BACKUPS -type f -name "*.tar.gz" -mtime +7 -exec rm {} \;
```


Es muy importante seguir el flujo de acciones en el orden que se especifican. 

Mientras realizamos operaciones específicas sobre la base de datos al trabajar con SQLite, si intentamos realizar una copia mientrar el servidor está escribiendo un nuevo mensaje, podríamos corromper por completo tanto la base de datos como la copia que estamos realizando. 

Por esa razón es muy importante cerrar el servicio momentáneamente, asumiendo que a las 3AM no debería de haber nadie utilizando la aplicación, y si es que lo hubiese simplemente sufriría una pequeña recarga.

Además para evitar que el disco de la máquina virtual con el tiempo de llene o se vaya agotando, se ha adaptado un sistema en el que se buscan y eliminan automáticamente cualquier backup que tenga más de 7 días de antigüedad. 

Este tiempo se ha definido un poco a ojo, pero siempre es posible modificar este script para cambiar este periodo de antigüedad.
