
Por defecto, Synapse crea los nombres de usuario en Matrix basándose en identificadores genéricos. En este caso el identificador que nos generaba lo hacía a partir de los correos electrónicos devueltos por el sistema de autentificación SSO. 

Por ejemplo el uid de una persona con un correo electrónico:`samuelcantero@correo.ugr.es` dentro del servidor se almacenaba como:

- `@samuelcantero=40correo.ugr.es:matrix.ugr.es`

Como podemos ver es bastante feo, largo e incomodo. Es por eso que se ha programado un Maping Provider personalizado que intercepta la respuesta que obtenemos del inicio de sesión de la UGR y formatea los nombres de usuario.

El id elegido será el la combinación de letras y/o números que aparecen en el correo antes del @. Añadiendo una distinción que identifica si estamos ante un correo de un alumno (correo.ugr.es) le pondremos al nombre de usuario un "e." y si estamos ante un correo que no lo sea (ugr.es) simplemente nos saltaremos esto. Quedaría de esta manera:

- alumo@correo.ugr.es -> @e.alumno:matrix.ugr.es
- profesor@ugr.es -> @profesor:matrix.ugr.es

El script de python tiene la siguiente forma:

``` python
import logging
from synapse.handlers.saml import DefaultSamlMappingProvider

logger = logging.getLogger(__name__)

class UgrSamlMappingProvider(DefaultSamlMappingProvider):
    def __init__(self, parsed_config, module_api):
        super().__init__(parsed_config, module_api)
        logger.warning("[UGR SAML] Mapeador UGR inicializado correctamente.")

    # Función que genera el vínculo con la base de datos
    def get_remote_user_id(self, saml_response, cliente_redirect_url):
        uid = saml_response.ava.get("uid", [None])[0]
        logger.warning(f"[UGR SAML] Vinculando usuario de la base de dato con uid: {uid}")

        if not uid:
           raise Exception("ERROR: Atributo uid no encontrado.")

        return uid

    # Función para crear el nombre de usuario en Matrix
    def saml_response_to_user_attributes(self, saml_response, failures, client_redirect_url):
        uid = saml_response.ava.get("uid",[None])[0]
        logger.warning(f"[UGR SAML] Generando nombre matrix para: {uid}")

        if not uid:
            raise Exception("ERROR: atributo uid no encontrado tras la traducción de PySAML2")

        email = uid.lower() # Lo pasamos todo a minúscula 
        nombre, dominio = email.split('@')

        if dominio == "ugr.es":
            nuevo_nombre = nombre # Para los profesores devolvemos el nombre antes del @ exclusivamente
        elif dominio == "correo.ugr.es":
            nuevo_nombre = f"e.{nombre}" # Para los alumnos devolvemos el nombre con "e.nombre" de antes del @
        else:
            raise Exception ("CUIDADO: se ha detectado un login con un correo con dominio distinto de 'ugr.es' o 'correo.ugr.es'")

        # Si el nombre ya existe, añadimos el número de fallos al final para evitar colisiones
        if failures > 0:
            nuevo_nombre = f"{nuevo_nombre}{failures}"

        logger.warning(f"[UGR SAML] Nombre asignado (mxid_localpart): {nuevo_nombre}")

        # Devolvemos el diccionario exacto que nos pide la documentación de Synapse
        return {
            "mxid_localpart": nuevo_nombre, # ID interno (ej. @e.pepe:matrix.ugr.es)
            "displayname": nuevo_nombre,  # Nombre visual en la aplicación
            "emails": [email]       # Vincula el correo real a la cuenta
        }
```

Cabe recalcar que este script no es un programa independiente, sino un módulo que se inyecta directamente en el código de Synapse. Para que funcione tenemos que interactuar con librerías y estructuras de datos muy específicas del protocolo. Para ello cargamos la clase que se encarga del manejo del MappingProvider con la siguiente línea:  

``` python
from synapse.handlers.saml import DefaultSamlMappingProvider
```


- **Herencia de `DefaultSamlMappingProvider`**: Obviamente no tenemos que programar un cliente SAML desde cero, es por ello que creamos nuestra propia clase `UgrSamlMappingProvider` que hereda de la oficial. Para ello no se nos puede olvidar inicializar correctamente la clase padre dentro del constructor, con los atributos que se nos especifican.
- **Método `get_remote_user_id`:** Este método es muy importante ya que es el encargado de capturar el valor de uid que recibimos desde el SAML de la universidad de Granada. Para ello hacemos uso del "Atribute Value Assertion" para capturar el valor exacto del uid. Además si esta función detecta que no se ha cargado este valor lanza un error con un Exception.

``` python        
uid = saml_response.ava.get("uid",[None])[0]
```

- **Restricción de los dominios:** El script además de modificar el nombre de usuario en la aplicación también actúa como cortafuegos, ya que solo acepta correos electrónicos con los dominios de "ugr.es" y "correo.ugr.es".
- **Sistema de logs:** Se ha intentado añadir un sistema de logs lo más descriptivo posible de tal manera que si hubiera algún tipo de error, se viese a través del monitoreo del servidor y así se pudiese identificar fácilmente que es lo que está fallando.
- **Atributos a devolver:** Synapse exige que la función de mapeo que hemos definido devuelva un diccionario con unas claves muy estrictas. Si estas variables no se devuelven exactamente en ese formato el motor de Matrix nos rechazaría la creación de la sesión. Estas claves son:

``` python
return {
	"mxid_localpart": nuevo_nombre, # ID interno (ej. @e.pepe:matrix.ugr.es)
	"displayname": nuevo_nombre,  # Nombre visual en la aplicación
	"emails": [email]       # Vincula el correo real a la cuenta
}
```
