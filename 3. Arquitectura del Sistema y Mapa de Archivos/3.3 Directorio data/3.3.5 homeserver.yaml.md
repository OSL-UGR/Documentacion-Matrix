
Este archivo es el que define la arquitectura de Synapse. Todas las decisiones importantes sobre como opera el servidor, a quién permite conectarse, cómo se integra con el exterior y que permisos tienen los usuarios se definen en este documento. Este tiene la siguiente estructura (he añadido unos cuantos comentarios al código para que queden más claro las distintas partes)


``` YAML
server_name: "matrix.ugr.es"
public_baseurl: "https://matrix.ugr.es/"
pid_file: /data/homeserver.pid

enable_cors: true

listeners:
  - port: 8008
    resources:
      - names: [client, federation]
        compress: false
    tls: false
    type: http
    x_forwarded: true
    bind_addresses: ['0.0.0.0']

database:
  name: sqlite3
  args:
    database: /data/homeserver.db

#Configuración de registros (Logs)
log_config: "/data/matrix.ugr.es.log.config"
media_store_path: /data/media_store

#Secreto compartido de registro (CENSURADO POR SEGURIDAD)
registration_shared_secret: "XXXXXXXX"
report_stats: false

password_config:
  enabled: false

saml2_config:
  brand_name: "Identificación UGR"
  sp_config:
    attribute_map_dir: "/data/attribute-maps"
    allow_unknown_attributes: true
    encryption_keypairs:
      - key_file: /data/saml.key
        cert_file: /data/saml.crt
    metadata:
      remote:
        - url: "url-de-metadatos.php-ugr"
    key_file: /data/saml.key
    cert_file: /data/saml.crt
    organization:
      name: "Universidad de Granada"
      display_name: "Matrix UGR"
      url: "https://matrix.ugr.es"
  
  user_mapping_provider:
    module: saml_mapping.UgrSamlMappingProvider
    config:
      mxid_source_attribute: "uid"
      displayname_source_attribute: "cn"

room_list_publication_rules:
  - user_id: "@usuario1:matrix.ugr.es"
    alias: "*"
    room_id: "*"
    action: allow
  - user_id: "@usuario2:matrix.ugr.es"
    alias: "*"
    room_id: "*"
    action: allow
  - user_id: "*"
    action: deny
```

Podemos dividir la estructura de este archivo en varias secciones:

1. **Variables iniciales:** 
	
	- `server_name`: Define el apellido de todos los usuarios (@usuario:matrix.ugr.es) además de que cuando el servidor arranca por primera vez este nombre se vuelve inmutable, cambiarlo corrompería toda la base de datos criptográfica. 
	- `public_baseurl:` Indica la url exacta por la cual se puede acceder al servidor.
	- `pid_file`: Simplemente le dice al SO donde guardar el archivo que contiene el identificador de proceso PID de Synapse en ejecución.
	- `enable_cors: true`: Activamos lo que se conoce como "Cross-Origin Resource Sharing". Lo que pasaba que sin poner esto las aplicaciones web (como nuestro Element Web) que se ejecutan en los navegadores de los usuarios, recibían un bloqueo de seguridad al intentar comunicarse con la API de Synapse.
	
2. **Listeners:** 
	
	- `port: 8008:` Es el puerto interno dobde trabaja la Api de Synapse.
	- `resources: [client, federation]:` Esto venía en la documentación, es para que atienda peticiones tanto de usuarios como de otros servidores externos.
	- `tls: false`: Desactiva el cifrado nativo de Synapse, delegando esta gestión a los certificados SSL y el HTTPS al Nginx.
	- `x_forwarded: true`: Esto es muy importante, hace que synapse acepte cabeceras externas de Ngynx para un correcto funcionamiento.
	- `bind_addresses: ['0.0.0.0']`: Permite que Synapse acepte tráfico de cualquier interfaz dentro de la red de Docker, permitiendo así que se pueda comunicar correctamente con Ngynx.
	
3. **Base de datos, clave e inicio de sesión:**
	
	- `database y media_store_path`: Definen las rutas de donde Synapse almacena el archivo homeserver.db además de las carpetas con imágenes audios o archivos enviados por los chats.
	- `registration_shared_secret`: Una clave criptográfica **QUE ES MUY IMPORTANTE** mantener oculta. Permite a los administradores registrar usuarios de emergencia a través de script o la terminal sin necesidad de pasar por el proceso web normal.
	- `password_config.enabled: false`: Quita el inicio de sesión normal con usuario y contraseña, forzando a que la única vía de entrada sea la pasarela de la UGR.
	
4. **Autentificación con saml2: (importante para el SSO)**
	
	- `metadata.remote.url`: Es la URL que nos proporcionó el CSIRC con las claves públicas de la UGR para verificar la autenticidad en los inicios de sesión.
	- `user_mapping_provider`: Carga el módulo que programamos con nuestro script de Python `saml_mapping.py`
	- `mxid_source_attribute` y `displayname_source_attribute`: Es para decirle al Synapse que variables concretas que llegan desde la UGR (el uid para el identificador y el cn para le nombre a mostrar) debe pasarle a nuestro script.
	
5. **Permisos de Directorio de Salas:**
	
	- Por defecto en Matrix cualquier usuario puede crear una sala y hacerla visiblemente pública en el directorio general de la organización. Esta característica está limitada a un número restringido de usuarios para evitar que cualquier usuario pueda mantener cientos de salas basura visibles para toda la comunidad. Synapse impedirá publicar salas como visibles en este directorio incluso si se tiene rango de Administrador General, solo podrá hacerlo los usuarios que se encuentren en ese listado:
	
	-  Las cuentas designadas (`@usuario1` y `@usuario2`) tienen el parámetro `action: allow`, otorgándoles el privilegio de publicar salas en el tablón general.
    
	- La regla  `- user_id: "*"` con `action: deny` bloquea esta opción para el resto.
